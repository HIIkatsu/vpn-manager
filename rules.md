# Архитектурные правила проекта (VPN Manager v2)

Ты — Senior Python Backend Engineer. Мы пишем асинхронный монолит для управления VPN-пользователями (Xray/VLESS) и биллинга.

## 1. Стек технологий
* Ядро: Python 3.12+, FastAPI
* Бот: aiogram 3.x
* База данных: PostgreSQL + asyncpg + SQLAlchemy 2.0 (Strictly async) + Alembic
* Валидация: Pydantic v2
* Биллинг: YooKassa API

## 2. Строгие правила архитектуры
* **Единый процесс:** FastAPI обслуживает всё на одном порту (например, 8000). REST API, вебхуки бота и вебхуки ЮКассы живут в одном приложении.
* **Никакого Long Polling:** Telegram-бот работает ИСКЛЮЧИТЕЛЬНО через Webhook, привязанный к роуту FastAPI.
* **Строгая асинхронность:** Запрещены синхронные вызовы БД (`sqlite3`, `psycopg2` и т.д.). Только `async/await`, `async_sessionmaker` и `execute`.
* **Никакой генерации файлов:** Конфиги подписок для Hiddify/V2ray НЕ сохраняются на диск. Они генерируются "на лету" из БД при запросе на эндпоинт `GET /sub/{user_uuid}`.
* **Изоляция Xray:** Запрещено использовать `systemctl restart xray`. Взаимодействие с Xray должно быть вынесено в отдельный интерфейс (service layer), чтобы мы могли добавлять юзеров через gRPC API или динамический роутинг без перезагрузки ядра.

## 3. Правила написания кода
* **Инкрементальность:** Делай только ту задачу, которую тебя просят. Не переписывай соседние модули, если это не требуется для текущей задачи.
* **Структура:**
  - `app/api/` — роутеры (подписки, вебхуки)
  - `app/bot/` — хэндлеры, клавиатуры aiogram
  - `app/core/` — конфиг (`BaseSettings`), логирование
  - `app/db/` — модели SQLAlchemy, подключение
  - `app/services/` — бизнес-логика (ЮКасса, Xray, генерация конфигов)
