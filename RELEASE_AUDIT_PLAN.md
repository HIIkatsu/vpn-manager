# Release Audit Plan — FastAPI + aiogram + SQLAlchemy + Xray VPN

## Цель документа

Этот документ фиксирует результаты аудита текущего состояния проекта и дает подробный, пошаговый план стабилизации перед выходом на рынок. Документ ориентирован на backend/system команду и должен использоваться как обязательный чек-лист перед production release.

---

## Executive Summary (Краткий вердикт)

Текущее состояние: **No-Go для market release**.

Основные причины:
1. Критические пробелы в безопасности платежного вебхука и админ-панели.
2. Операционная хрупкость рантайма: бот polling и фоновые воркеры исполняются внутри FastAPI-процесса.
3. Недостаточно жесткая модель идемпотентности и управления транзакциями для денежных операций.
4. Архитектурная перегрузка `main_app.py`, мешающая масштабированию, тестируемости и безопасному развитию.

Условие перехода к ограниченному beta rollout:
- Полное закрытие блока **Шаг 1 (Security blockers)**.
- Декомпозиция рантайма по процессам минимум по **Шагу 2**.

Условие перехода к полноценному release:
- Закрытие шагов 1–7 с подтверждающими тестами, метриками и runbook-ами.

---

## Обнаруженные проблемы и риски

### [КРИТИЧНО] Red Flags (исправлять до релиза)

#### RF-1: Фактическое отсутствие реальной защиты вебхука YooKassa

**Проблема:**
- Проверка IP сделана заглушкой, которая всегда разрешает запросы.
- Rate limiter сделан заглушкой, которая всегда пропускает трафик.
- Логика проверки webhook auth в сервисе существует, но не применяется в endpoint.

**Почему это опасно:**
- Любой внешний источник может отправлять поддельные события на платежный endpoint.
- Возможны попытки мошеннических активаций, DDoS на billing-обработчик и отказ в обслуживании легитимных событий.

**Что нужно сделать:**
- Включить обязательную валидацию webhook auth по официальной схеме YooKassa.
- Включить реальный rate-limit на endpoint `/webhook/yookassa`.
- Включить строгую проверку IP/CIDR allowlist (если используется как дополнительный периметр).
- Добавить мониторинг отказов по причине auth/ip/rate-limit.

**Критерий готовности:**
- Поддельные payload/headers не проходят в бизнес-логику.
- Неавторизованный трафик получает 401/403/429 с корректной причиной.
- В метриках есть раздельные счетчики reject-ов.

---

#### RF-2: Риск неконсистентных удалений/деактиваций пользователей

**Проблема:**
- Воркер удаления завязан на комбинацию `is_active == False` и устаревшую `sub_end_date`.
- В админке флаг `is_active` меняется до фактического применения действия в Xray.
- Это создает промежуточные неконсистентные состояния между БД и Xray.

**Почему это опасно:**
- Ошибочное удаление профиля пользователя, который должен быть активным.
- Потеря данных конфигурации и репутационные/финансовые потери.

**Что нужно сделать:**
- Ввести явную state-модель жизненного цикла пользователя (например: `active`, `suspended`, `expired`, `pending_delete`).
- Разделить намерение админа (pending action) и фактическое примененное состояние.
- Воркеру удаления дать работу только по однозначным “подтвержденным” статусам.

**Критерий готовности:**
- Удаление возможно только при достижении формального конечного статуса.
- Нет переходов статуса “вперед-назад” без явного журнала.

---

#### RF-3: Денежный workflow недостаточно устойчив к конкурентным сценариям

**Проблема:**
- Есть row lock и статусная модель, но внешние side effects (Xray вызовы) выполняются в уязвимом транзакционном контуре.
- Одни и те же платежи могут обрабатываться из разных источников: webhook, ручной форс-чек из бота, фоновые проверки pending.

**Почему это опасно:**
- При сетевых деградациях и ретраях возможны сложные “полу-состояния” (внешняя система обновилась, БД не зафиксировала факт/или наоборот).
- Повышается риск повторной обработки или некорректных retry.

**Что нужно сделать:**
- Установить единый “источник истины”: webhook-first.
- `process_pending` оставить как компенсационный механизм с жесткими правилами.
- Зафиксировать единую политику идемпотентности для платежных событий: event dedup, retry policy, timeout policy.

**Критерий готовности:**
- При дубликатах webhook нет двойных начислений и двойных продлений.
- При сбоях Xray/БД есть предсказуемый compensating path.

---

#### RF-4: Дефолтные admin credentials

**Проблема:**
- Есть значения по умолчанию для admin username/password.

**Почему это опасно:**
- При ошибке конфигурации сервис может подняться с предсказуемыми credentials.

**Что нужно сделать:**
- Запретить старт приложения при дефолтных/пустых admin secret.
- Перенести секреты в защищенный secret-store/CI variables.

**Критерий готовности:**
- Приложение не стартует, если admin creds небезопасны.

---

### [АРХИТЕКТУРА] Yellow Flags

#### YF-1: Перегрузка `main_app.py`

**Проблема:**
- В одном файле смешаны API, startup/shutdown, воркеры, orchestration бота, админка, webhook биллинга.

**Последствия:**
- Рост стоимости изменений.
- Усложнение изоляционного тестирования.
- Повышенный риск регрессий при “горячих фиксах”.

**Решение:**
- Разрезать по bounded contexts: routers, lifecycle, services, worker entrypoints.

---

#### YF-2: Бот polling и worker внутри web-процесса

**Проблема:**
- Polling и фоновые циклы стартуют в FastAPI startup.

**Последствия:**
- При multiprocess/replicas — дубли воркеров и polling, гонки, двойные действия.
- Плохая управляемость shutdown/restart.

**Решение:**
- Разнести по отдельным процессам/сервисам (api, bot, workers).

---

#### YF-3: Разъезд DI и точек управления сессией

**Проблема:**
- Сессии создаются несколькими способами, DI в разных местах не унифицирован.

**Последствия:**
- Труднее гарантировать единый transaction boundary.
- Сложнее внедрить централизованную rollback/commit policy.

**Решение:**
- Ввести единый composition root и единый UoW pattern для приложения.

---

#### YF-4: Слабая гигиена модели данных

**Проблема:**
- `PendingAction` расположен в файле с `User`; имеются дубли import-блоков.

**Последствия:**
- Сигнал неустойчивой структуры доменной модели.
- Риск ошибок при масштабировании моделей/миграций.

**Решение:**
- Перенести модели в отдельные файлы/модули с четкой ответственностью.

---

## Подробный пошаговый план рефакторинга

## Шаг 1 — Security Blockers (P0, до релиза)

**Цель:** закрыть все прямые векторы несанкционированного доступа к платежам и админке.

### 1.1 Webhook auth hardening
- В endpoint вебхука сделать обязательную проверку авторизации.
- Не обрабатывать payload до прохождения auth.
- Любая ошибка авторизации должна быть явной и метрифицируемой.

### 1.2 IP allowlist hardening
- Реализовать корректную проверку CIDR/IP через валидный parser.
- Учитывать доверенную цепочку reverse proxy (если есть).
- Запретить fallback “unknown -> allow”.

### 1.3 Rate limiting
- Включить production rate limiter (in-memory/redis based в зависимости от архитектуры).
- Ввести лимиты отдельно на webhook и admin auth.

### 1.4 Admin perimeter
- Запретить дефолтные креды (fail-fast).
- Ограничить `/admin` сетевым периметром (ingress/Nginx/WAF).
- Добавить лог аудита входов.

### Артефакты выхода шага
- Security checklist signed-off.
- Набор тестов негативных сценариев (поддельные события).
- Dashboard метрик отклонений.

---

## Шаг 2 — Runtime Decomposition (P0/P1)

**Цель:** убрать хрупкую совместную жизнь API + bot polling + background loops.

### 2.1 Разделить процессы/entrypoints
- Процесс `api`: только FastAPI.
- Процесс `bot`: только aiogram polling/webhook handler.
- Процесс `worker`: периодические задачи (expiry/pending/reconcile).

### 2.2 Управление lifecycle
- Для каждого процесса: собственные startup/shutdown hooks.
- Явный graceful shutdown с лимитами времени и cleanup.

### 2.3 Гарантия single-run для критичных воркеров
- Leader election или distributed lock для задач, которые не должны идти параллельно.

### Артефакты выхода шага
- Обновленный docker-compose/systemd/k8s manifests.
- Runbook “Restart/Recovery”.
- Нулевой duplicate-run в staging под scale-test.

---

## Шаг 3 — Расслоение архитектуры приложения (P1)

**Цель:** восстановить SoC и тестируемость.

### 3.1 Разнести API по роутерам
- `billing_router` (вебхуки/платежи)
- `subscription_router`
- `admin_router`
- `health_router`

### 3.2 Вынести lifecycle и orchestration
- `lifespan`/runtime модуль для чистого старта API без побочных функций.

### 3.3 Application service слой
- Endpoint должен быть тонким: validate -> call service -> map response.
- Вся бизнес-логика платежей и подписок — в сервисах use-case уровня.

### Артефакты выхода шага
- Архитектурная схема модулей.
- Контрактные тесты по каждому роутеру.

---

## Шаг 4 — Идемпотентность и Race-устойчивость billing (P0/P1)

**Цель:** гарантировать отсутствие двойных начислений/продлений при дублях и конкурентных событиях.

### 4.1 Единый источник активации
- Основной путь: webhook payment.succeeded.
- `process_pending` — только компенсация с ограниченной частотой и telemetry.

### 4.2 Стандартизовать event dedup
- Формализовать ключ идемпотентности.
- Логировать все конфликтные/дублирующие события.

### 4.3 Транзакционный контур
- Минимизировать время удержания блокировки.
- Явно разделить DB-state update и внешние side effects.
- Добавить recovery-механику на случай “внешняя операция прошла, а commit нет”.

### Артефакты выхода шага
- Документ “Billing idempotency contract”.
- Chaos-тест сценариев duplicate webhook + network failure.

---

## Шаг 5 — Единая transaction/session policy (P1)

**Цель:** убрать разрозненное управление commit/rollback и сделать поведение предсказуемым.

### 5.1 Единый Unit of Work
- Стандарт “один use case — один transaction boundary”.
- Явные правила commit/rollback в application layer.

### 5.2 Исключения и rollback policy
- Ошибки домена и инфраструктуры должны иметь единые категории.
- rollback должен происходить одинаково для всех handler’ов.

### 5.3 Session lifecycle в боте и API
- Унификация DI и сессий между FastAPI и aiogram middleware.

### Артефакты выхода шага
- Технический RFC по transaction policy.
- Интеграционные тесты с форсированными исключениями.

---

## Шаг 6 — Domain model hygiene (P2)

**Цель:** структурная чистота моделей и миграций.

### 6.1 Нормализовать расположение моделей
- Каждая модель в своем модуле.
- Убрать дубли импортов и неочевидные зависимости.

### 6.2 Timezone discipline
- Везде UTC-aware datetime.
- Запрет смешения naive/aware дат.

### Артефакты выхода шага
- Линтер/статический чек на datetime policy.
- Миграции проверены на staging snapshot.

---

## Шаг 7 — Observability, SRE и эксплуатационная готовность (P1)

**Цель:** видеть проблемы до того, как их увидят пользователи.

### 7.1 Structured logs
- Поля: `request_id`, `payment_id`, `telegram_id`, `event_id`, `action_source`.

### 7.2 Метрики
- webhook reject by reason
- payment activation latency
- duplicate event count
- pending queue size
- worker lag

### 7.3 Alerting
- Алерт “payment succeeded but activation failed”.
- Алерт на рост duplicate/429/403.

### 7.4 Runbooks
- “Что делать при застрявших pending-платежах”.
- “Что делать при сбое Xray”.
- “Как безопасно перезапустить worker/bot/api”.

### Артефакты выхода шага
- Дашборды и alert rules.
- Эксплуатационные runbook-документы.

## Критерии готовности к релизу (Definition of Ready for Launch)

1. Security:
   - Нет заглушек в критических проверках.
   - Admin perimeter закрыт, дефолтных секретов нет.

2. Billing correctness:
   - Идемпотентность подтверждена тестами конкурентного доступа.
   - Нет двойных начислений в повторных webhook сценариях.

3. Runtime stability:
   - API, bot, worker разделены.
   - Нет duplicate execution при scale-out.

4. DB safety:
   - Единая transaction policy внедрена.
   - Ошибки не оставляют “грязных” промежуточных состояний.

5. Operations:
   - Метрики, алерты, runbooks в наличии и проверены.

- **QA Lead:** негативные и конкурентные сценарии, регрессионный набор.

## Финальный вывод

Проект можно довести до production-grade состояния, но в текущем виде выпуск на рынок несет высокий риск финансовых и эксплуатационных инцидентов. Этот план обязателен к исполнению как pre-release stabilization roadmap.
